# 上下文对话与折叠轮次界面设计

## 1. 目标

在现有 Buildroom、FastAPI、LangGraph、DeepSeek 和 RAG 架构上增加真正可连续补充信息的对话体验：

- 寒暄消息由大模型语义判断并亲切回复，同时自然引导用户讨论装机。
- 题外消息由大模型判断并友好说明能力范围，不进入预算、检索和选型节点。
- 装机信息不足时，先根据已经明确的信息给出有用的初步判断，再提出最少且具体的补充问题。
- 用户下一轮补充内容时，Agent 能看到本会话近期上下文，而不是把补充内容当成孤立的新需求。
- 前端将每轮用户消息和 Agent 回复保存为独立、可折叠的对话卡片；最新一轮默认展开，历史轮次默认折叠。
- 桌面端左侧栏和主对话区之间的分隔线可由用户拖动，页面各区域随侧栏宽度实时自适应。
- “硬件新讯”列表的型号、来源状态和价格使用清晰可读的字号与行高。

## 2. 当前约束

- 当前任务运行时仍由 `InMemoryTaskRepository` 保存；本次不引入新的 MySQL 会话表或迁移。
- 每一次用户提交仍创建一个独立推荐任务和独立 Agent trace。
- 会话上下文由前端随请求发送，适合当前 localhost 单用户开发场景。
- 不展示模型隐藏思维链。界面只展示可审计的节点状态、输入输出摘要、工具调用、证据和耗时。
- 保留现有 Buildroom 视觉语言、左侧栏、硬件目录、Agent 轨迹和方案结果组件。

## 3. 对话数据模型

### 3.1 API 请求

`RecommendationRequest` 增加 `context_messages`：

```json
{
  "text": "预算大约一万元",
  "context_messages": [
    {"role": "user", "content": "我想配一台主要玩 2K 游戏的电脑"},
    {"role": "assistant", "content": "可以先按 2K 游戏主机规划。你的预算是多少，是否需要显示器？"}
  ]
}
```

只允许 `user` 和 `assistant` 角色；最多携带最近 12 条消息；单条和总长度均受 Pydantic 校验。当前 `text` 永远代表最新用户消息。

### 3.2 任务状态

任务状态增加：

- `response_kind`: `recommendation | clarification | casual | off_topic | error`
- `assistant_message`: 面向用户的完整自然语言回复

`assistant_message` 与 Agent trace 分离。前者用于聊天，后者用于审计。

### 3.3 前端轮次

前端使用 `ConversationTurn[]`，每轮至少保存：

- 本轮 ID、任务 ID、用户原文和创建时间。
- 状态、进度、Agent runs 和流式片段。
- `assistantMessage`、澄清问题、最终推荐结果或错误。
- `expanded` 折叠状态。

输入框使用独立的 `draft` 状态，提交后立即清空，不再覆盖历史用户消息。

## 4. 后端行为

### 4.1 意图分类

`IntentClassificationAgent` 继续真实调用 DeepSeek，但扩展结构化输出：

- `pc_build`、`pc_upgrade`、`hardware_consultation`：继续执行装机图。
- `casual`：寒暄、感谢、情绪表达等不要求装机结论的社交消息。
- `off_topic`：与装机和桌面硬件无关的知识或任务。
- `assistant_reply`：当不继续装机图时，由模型生成亲切、简短且带装机引导的回复。

确定性发布门只检查结构是否合法，不用关键词列表代替语义裁决。`casual` 和 `off_topic` 在意图节点结束，后续节点标记 `skipped`，任务状态使用 `completed`，`response_kind` 分别为 `casual` 或 `off_topic`。

### 4.2 信息不足

`RequirementAgent` 接收当前消息、近期上下文和不可覆盖的预算交接结果。`RequirementAnalysis` 增加 `partial_answer`：

- 概括已经确认的预算、用途、分辨率、已有配件或外设范围。
- 对已经明确的要求给出安全的方向性建议，不伪造具体采购清单或实时价格。
- 只提出会实质改变配置的补充问题。

确定性完整性检查仍负责保证预算和主要用途等硬信息不能被模型错误忽略。若需要澄清，工作流设置 `response_kind=clarification`，将 `partial_answer` 与去重后的问题组合成 `assistant_message`，然后跳过检索和选型节点。

### 4.3 多轮上下文

- Supervisor、Intent、Budget 和 Requirement 节点接收同一份经过长度限制的对话上下文。
- 当前消息中的明确修正优先于历史消息，例如“不是 8000，是 1 万”。
- Budget Agent 的确定性解析先处理最新消息；最新消息没有预算时才从历史用户消息恢复预算证据。
- Requirement Agent 在语义合并时读取上下文，补全上一轮已确认的用途、外设和已有配件。
- Search、Selection、Compatibility 和 Report 只读取合并后的结构化 profile，不直接依赖前端消息格式。

## 5. 前端交互

- 对话区域按时间顺序渲染轮次折叠卡片，使用原生语义化 `details/summary` 或等价受控组件。
- 卡片标题展示用户消息摘要、本轮状态、时间和展开箭头。
- 展开内容依次展示用户消息、Agent 头像与流式回复、澄清提示或完整方案。
- 新轮次创建时只展开最新轮次，旧轮次自动折叠；用户可以手动重新展开任意历史轮次。
- 右侧 Agent 轨迹显示当前选中轮次；点击某一轮卡片时同步切换轨迹。
- 每轮流式更新只写入对应 `turnId`，避免并发状态覆盖其他历史轮次。
- 失败的轮次保留在对话中并显示可读错误，不清空已经完成的历史结果。
- 新建对话清空当前轮次集合；左侧最近任务功能继续按任务读取单轮结果。

### 5.1 可调整侧栏

- 桌面端在侧栏右边缘提供明确但克制的拖拽手柄，使用 Pointer Events 统一支持鼠标、触控板和触控笔。
- 拖动时更新根布局 CSS 变量 `--sidebar-width`，而不是分别写死侧栏、主区和输入框的像素位置。
- 侧栏宽度限制在 `260px` 到 `460px`，同时不得超过视口宽度的 40%，避免用户把主对话区挤没。
- 用户选择的宽度写入 `localStorage`，刷新页面后恢复；非法或过期值自动回落到默认宽度。
- 手柄支持键盘操作：获得焦点后，左右方向键按固定步长调整宽度，并提供可读的 `aria-label`、`role=separator`、当前值和最小最大值。
- 双击手柄恢复默认宽度。拖动期间禁止文本选择并显示列调整光标，释放或失焦后恢复。
- 主布局、固定输入框和打开 Agent 轨迹后的内容宽度都从同一 CSS 变量计算，不能继续保留互相矛盾的 `292px`、`316px` 等常量。
- 视口小于 `760px` 时关闭桌面拖拽能力，继续使用现有移动端抽屉侧栏。

### 5.2 硬件新讯可读性

- 硬件型号正文桌面端不小于 `12px`，价格不小于 `11px`，来源与质量状态不小于 `9px`。
- 列表项增加行高和垂直间距，长型号最多显示两行，不再用单行极小字强制截断。
- 分类标签保持紧凑但不低于 `10px`；选中态继续沿用现有薄荷绿视觉层级。
- 侧栏变宽时硬件文字区域自然获得更多空间；侧栏变窄时使用两行省略，不产生横向滚动。

## 6. SSE 与结果处理

- SSE `status` 事件携带 `response_kind` 和 `assistant_message`。
- `casual`、`off_topic` 和 `clarification` 可没有推荐结果，但必须有聊天回复。
- 推荐完成时仍发送现有方案流式片段和 `result`。
- 前端流式类型增加聊天文本片段；如果后端暂时只在终态得到完整回复，前端以段落渐进显示，不伪造 Agent 执行进度。

## 7. 测试设计

### 后端

- 寒暄由假 LLM 判定为 `casual`，返回亲切回复，预算及之后节点全部跳过。
- 题外消息返回 `off_topic`，不触发 RAG、SerpAPI 或硬件选型。
- 已给预算但用途不明确时，返回包含预算初步建议和用途追问的 `assistant_message`。
- 已给用途但未给预算时，返回用途方向建议和预算追问。
- 第二轮只补充预算时，历史用途能够进入合并 profile。
- 当前轮修正预算时覆盖历史预算。
- API status 与 SSE 正确输出 `response_kind` 和 `assistant_message`。

### 前端

- 类型检查和生产构建通过。
- 连续提交两轮后，两张轮次卡片都存在，第一轮内容不会被第二轮覆盖。
- 最新轮默认展开，旧轮可重新展开。
- 澄清、寒暄、题外和推荐四种回复都使用同一轮次组件正确显示。
- 切换轮次时右侧 Agent trace 与该轮一致。
- 拖动侧栏时主对话、固定输入框和 Agent 轨迹没有重叠或跳变，刷新后宽度能够恢复。
- 分隔线可通过键盘左右键调整并可双击复位。
- 在最小侧栏宽度下，硬件型号、来源和价格仍达到规定字号且不产生横向滚动。

## 8. 非目标与后续工作

- 本次不实现跨设备、跨浏览器或服务重启后的完整会话恢复。
- 本次不将聊天消息写入 MySQL；后续可新增 `conversations` 和 `conversation_messages` 表实现服务端持久化。
- 本次不允许模型绕过预算、兼容性和功耗硬校验。
