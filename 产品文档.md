# AI装机多智能体系统产品文档

## 1. 产品定位

AI装机多智能体系统面向有装机需求的普通用户、游戏玩家和轻度创作者，帮助用户在预算约束下获得兼容、可解释、带实时价格参考的整机配置方案。

## 2. 目标用户

1. 不懂硬件但想快速得到靠谱配置的新手用户。
2. 有明确预算，希望最大化性价比的游戏玩家。
3. 需要兼顾游戏、剪辑、直播等复合场景的内容创作者。
4. 电脑店、装机博主、社群顾问等半专业用户。

## 3. 核心问题

传统装机决策存在以下问题：

1. 用户不知道该如何拆预算。
2. 单看参数难以判断真实搭配是否合理。
3. 配件之间兼容关系复杂，容易踩坑。
4. 市场价格波动大，用户难以判断是否溢价。
5. 推荐结果常常只有结论，没有可解释理由。

## 4. 产品目标

1. 输入自然语言需求，输出结构化装机建议。
2. 自动完成核心硬件选择、兼容校验、预算控制。
3. 尽可能结合实时价格和评测信息，提高可信度。
4. 输出清晰、可解释、可对比的推荐报告。

## 5. MVP范围

### 5.1 支持范围

1. 支持台式机主机配置推荐。
2. 支持预算、用途、分辨率、机箱尺寸等需求输入。
3. 支持 CPU、GPU、主板、内存、存储、电源、散热、机箱的推荐。
4. 支持价格查询、兼容校验、预算分配、推荐理由生成。
5. 支持主推荐方案 + 备选方案。

### 5.2 暂不纳入首版

1. 电竞椅、桌搭等非核心外设。
2. 复杂的二手市场价格分析。
3. 多地区多电商深度比价。
4. 自动下单与电商联动。

## 6. 用户流程

1. 用户输入需求，例如“6500预算，2K游戏，机箱要小一点，不要太吵”。
2. 系统解析缺失项，必要时发起少量追问。
3. Agent 组合生成候选配置。
4. 系统进行价格补充、兼容校验和评分。
5. 系统返回主推荐方案、备选方案和解释报告。

## 7. 核心功能

### 7.1 需求解析

将自然语言转成结构化字段：

1. 预算
2. 使用场景
3. 分辨率/游戏类型
4. 是否包含外设
5. 机箱尺寸偏好
6. 品牌偏好
7. 噪音与升级偏好

### 7.2 候选硬件生成

按预算和用途生成每类硬件的候选项，并组合成整机方案。

### 7.3 实时搜索与知识补充

基于外部搜索与本地知识库补充：

1. 实时价格
2. 评测摘要
3. 历史搭配案例

### 7.4 兼容校验

至少覆盖：

1. CPU 与主板插槽匹配
2. GPU 长度与机箱空间
3. 散热器与机箱支持
4. 电源功率是否满足冗余要求
5. 内存代际与主板支持

### 7.5 评分与解释

输出五个维度分数：

1. 性能匹配度
2. 预算匹配度
3. 兼容稳定性
4. 价格合理性
5. 升级空间

### 7.6 报告生成

最终报告至少包含：

1. 主推荐配置
2. 备选配置
3. 核心推荐理由
4. 价格说明
5. 兼容性结论
6. 风险提示
7. 升级建议

## 8. 非功能要求

1. 单次推荐响应时间目标：`10-25s`
2. 系统应支持缓存，避免重复搜索
3. 所有外部检索需支持失败降级
4. 每个任务需保留执行日志，便于审计和排障
5. 最终结果应标识数据时间和实时性等级

## 9. 数据库设计建议

### 9.1 需要什么样的数据库

本项目首版最适合使用本地 `MySQL 8.x` 作为主数据库。

原因：

1. 适合存放结构化硬件规格数据。
2. 适合存放任务、结果、日志等关系型数据。
3. 资料多，适合初学者，后续也容易迁移到服务器或云数据库。
4. 与 `FastAPI + SQLAlchemy` 或 `pymysql` 配合成熟。

### 9.2 为什么不能只靠大模型

这个系统不只是聊天，它还要处理：

1. 硬件规格查询
2. 兼容关系校验
3. 价格快照留存
4. 任务状态跟踪
5. 推荐结果存档
6. Agent 运行轨迹审计

这些都需要稳定的数据持久化，不能只放在内存里，也不能每次都让模型重新“猜”。

### 9.3 本地 MySQL 连接方式

当前项目建议直接使用本地 MySQL：

```text
mysql-url = mysql+pymysql://root:123456@127.0.0.1:3306/diy_agents
```

其中：

1. `root` 是数据库用户名。
2. `123456` 是数据库密码。
3. `127.0.0.1` 表示连接本机数据库。
4. `3306` 是默认 MySQL 端口。
5. `diy_agents` 是项目数据库名。

### 9.4 首版建议的核心表

首版建议至少建立以下 7 张表：

1. `hardware_parts`
2. `compatibility_rules`
3. `recommendation_tasks`
4. `recommendation_results`
5. `price_snapshots`
6. `agent_runs`
7. `tool_call_logs`

### 9.5 各表用途

#### `hardware_parts`

存放 CPU、GPU、主板、内存、硬盘、电源、散热、机箱等基础规格。

关键字段建议：

1. `part_type`
2. `brand`
3. `model`
4. `socket`
5. `memory_type`
6. `tdp_watt`
7. `gpu_length_mm`
8. `cooler_height_mm`
9. `psu_watt`
10. `price_reference`

#### `compatibility_rules`

存放兼容规则，避免全部逻辑硬编码。

关键字段建议：

1. `rule_type`
2. `left_part_type`
3. `right_part_type`
4. `rule_expression`
5. `severity`
6. `message_template`

#### `recommendation_tasks`

记录一次用户装机请求及处理进度。

关键字段建议：

1. `task_id`
2. `user_input`
3. `structured_requirement_json`
4. `status`
5. `current_stage`
6. `progress`
7. `need_realtime_price`
8. `need_alternative_build`

#### `recommendation_results`

记录任务最终输出的推荐结果。

关键字段建议：

1. `task_id`
2. `summary`
3. `primary_build_json`
4. `alternative_build_json`
5. `compatibility_passed`
6. `score_breakdown_json`
7. `price_freshness`
8. `risks_json`
9. `upgrade_suggestions_json`

#### `price_snapshots`

记录硬件型号在某一时间点的价格快照。

关键字段建议：

1. `part_type`
2. `brand`
3. `model`
4. `source`
5. `price`
6. `currency`
7. `captured_at`
8. `raw_payload_json`

#### `agent_runs`

记录每个 Agent 的执行轨迹。

关键字段建议：

1. `task_id`
2. `agent_name`
3. `status`
4. `iterations`
5. `input_summary`
6. `output_summary`
7. `started_at`
8. `finished_at`

#### `tool_call_logs`

记录工具调用日志。

关键字段建议：

1. `task_id`
2. `agent_name`
3. `tool_name`
4. `request_payload_json`
5. `response_summary`
6. `latency_ms`
7. `result_status`
8. `created_at`

### 9.6 首版为什么不建议只建一张大表

如果把所有数据混在一张表里，会出现：

1. 任务数据和硬件数据耦合太重。
2. 一次任务的多轮执行过程难以表达。
3. 推荐结果、价格快照和工具日志更新时间不同，不适合强行混合。

因此，首版最合适的是“硬件表 + 任务表 + 结果表 + 日志表”的分层结构。

## 11. 成功指标

1. 推荐结果生成成功率
2. 兼容校验准确率
3. 用户点击“接受推荐/收藏方案”比例
4. 用户二次追问率
5. 外部搜索调用成本

## 12. 版本建议

### V1

聚焦主机推荐闭环，验证“Agent化装机推荐”是否成立。

### V2

增加外设、更多价格渠道、更多品牌偏好和升级策略。

### V3

支持账号体系、配置收藏、分享、历史报告与个性化推荐。
