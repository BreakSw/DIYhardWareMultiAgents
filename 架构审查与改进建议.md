# AI装机多智能体系统架构审查与改进建议

## 1. 当前项目判断

当前仓库仅包含一份需求/方案文档 `agent(1).docx`，尚未包含前端、后端、数据层、Agent编排或工具层代码。因此本次审查基于方案文档而非实现代码，结论应视为“架构级预审”。

## 2. 方案亮点

1. 业务目标明确：围绕“装机推荐 + 兼容校验 + 实时价格/评测检索 + 报告生成”构建闭环，价值清晰。
2. Agent分工完整：需求解析、检索、硬件筛选、适配校验、打分、报告合成的职责链条基本完整。
3. ReAct使用场景合理：适合处理“信息缺失、需要多轮搜索、需要多次兼容修正”的装机决策链路。
4. 工具化思路正确：把价格检索、兼容校验、预算拆分、RAG检索拆成独立工具，利于后续替换和扩展。
5. 降级意识较强：已经考虑 SerpAPI 超时、限流、欠费等失败场景，说明方案具备一定工程意识。

## 3. 主要问题

### 3.1 Agent数量偏多，早期实现成本过高

文档把系统拆成 18 个 Agent，这在方案层面看起来完整，但在第一阶段落地时会带来明显问题：

1. 编排复杂度高，排障困难。
2. 各 Agent 间上下文传递成本高。
3. Token 消耗和工具调用次数会迅速膨胀。
4. 很多职责实际可以先合并，否则容易出现“概念完整、实现空转”。

建议首版收敛为 6 个核心角色：

1. `SupervisorAgent`
2. `RequirementAgent`
3. `HardwareSelectionAgent`
4. `SearchAndKnowledgeAgent`
5. `CompatibilityAndPricingAgent`
6. `ReportAgent`

### 3.2 “ReAct everywhere” 需要边界，不宜全链路无限循环

文档默认所有 Agent 都采用多轮 ReAct，这是理念上合理、工程上危险的设计。若缺少约束，会出现：

1. 同一问题重复搜索。
2. 相邻 Agent 重复做相同判断。
3. 单任务延迟不可控。
4. SerpAPI 调用费用不可控。

建议增加统一约束：

1. 单 Agent 最大思考轮次：`3`
2. 单 Agent 最大外部工具调用次数：`5`
3. 单任务最大 SerpAPI 调用次数：`8`
4. 同类搜索命中缓存后禁止再次发起同关键词检索
5. 出现连续两次无增量结果时直接退出当前循环

### 3.3 数据模型没有定义清楚

文档描述了很多工具，但没有统一对象模型。没有统一模型，后续会出现 Agent 输出难以复用的问题。

至少需要先定义这些核心对象：

1. `UserRequirement`
2. `HardwareCandidate`
3. `BuildDraft`
4. `CompatibilityIssue`
5. `PriceSnapshot`
6. `BuildScore`
7. `RecommendationReport`

### 3.4 评分体系不够可解释

“性价比打分 Agent” 是关键卖点，但文档没有说明分数如何计算。若缺少可解释性，用户很难信任推荐结果。

建议分数拆成五个维度：

1. 性能匹配度
2. 预算匹配度
3. 兼容稳定性
4. 价格合理性
5. 可升级空间

每项给出权重、原始依据、扣分原因和替代建议。

### 3.5 缺少任务状态机和审计机制

该系统属于高工具调用、多阶段推理流程，必须具备可观测性。当前文档未说明：

1. 每个 Agent 的输入输出如何落日志。
2. 某次推荐为何选择 A 而非 B。
3. 外部价格数据来自何时、何渠道。
4. 某轮兼容校验为什么失败。

建议补充：

1. `task_run` 表记录任务总状态。
2. `agent_run` 表记录每个 Agent 的执行轨迹。
3. `tool_call_log` 表记录每次工具调用参数、耗时、结果摘要。
4. `report_evidence` 表记录报告中的评分证据、价格证据和评测证据。

## 4. 推荐落地架构

## 4.1 分层

1. 前端层：`React 18 + TypeScript + Ant Design`
2. API层：`FastAPI`
3. 编排层：`LangGraph Supervisor`
4. Agent层：收敛后的 6 个核心 Agent
5. 工具层：价格搜索、规格查询、兼容计算、RAG、缓存
6. 数据层：`MySQL + Redis + Qdrant`

## 5. MVP建议

首版不要一次实现完整 18 Agent，建议按下面范围上线：

1. 只支持主机 6 大件：CPU、GPU、主板、内存、硬盘、电源。
2. 先不做外设和电竞椅。
3. 先支持单套推荐 + 单套备选推荐，不做过多组合。
4. SerpAPI 只做价格和评测摘要，不做趋势分析。
5. 报告只输出：配置单、预算占比、兼容校验结果、价格来源、推荐理由。

## 6. 审查结论

这是一个方向正确、卖点清晰、适合 Agent 化实现的项目方案，但当前文档更像“概念完备版”，还没有进入“可直接开发版”。

如果按原文直接开工，最大风险不是技术不可做，而是：

1. 角色拆分过细导致编排过重。
2. ReAct 循环缺少上限导致成本失控。
3. 数据结构未统一导致 Agent 无法稳定协作。
4. 评分与推荐理由不可解释，影响产品可信度。

建议先以“6 Agent + 明确数据模型 + 有界 ReAct + 可审计日志 + MVP范围收敛”的方式推进，这样更容易在第一版做出稳定成果。
